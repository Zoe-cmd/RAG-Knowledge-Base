## 1. 项目概述

### 1.1 项目背景

本项目是一个面向**个人学习与作品集展示**的 AI 应用。开发者希望通过从零构建一个完整的 RAG（Retrieval-Augmented Generation）文档知识库，掌握 AI 应用开发的全流程：文档解析、文本切分、向量化、向量检索、大模型调用、前后端工程化。

项目采用 **MVP（最小可行产品）** 思路，第一版只实现核心闭环功能，严格控制复杂度，确保初级开发者可独立完成。后续版本再逐步扩展高级特性。

### 1.2 项目目标

#### 业务目标

1. 用户能够上传 PDF、Word(docx)、Markdown、TXT 四类文档
2. 系统自动解析、切分文档并生成向量存入本地向量库
3. 用户能够基于已上传文档进行自然语言提问，获得带引用来源的 AI 回答
4. 系统保存聊天历史，用户可回看历史问答

#### 技术目标

1. 跑通完整 RAG 闭环：上传 → 解析 → 切分 → Embedding → 检索 → LLM 生成
2. 形成清晰可读、可作为作品集展示的代码工程
3. 代码结构支持后续扩展（多知识库、多用户、插件等留有扩展点但不实现）

#### 成功标准

| 编号 | 成功标准 | 验证方式 |
|------|----------|----------|
| S1 | 可上传 4 类文档并完成解析切分向量化 | 实际上传各类文档，检查 Chroma 中向量数 |
| S2 | 提问可基于文档内容回答并标注引用来源 | 实际提问，核对回答与引用片段一致 |
| S3 | 聊天历史可保存与回看 | 多轮对话后刷新页面，历史仍在 |
| S4 | 项目可一键本地启动运行 | 按 README 步骤可在干净环境启动 |
| S5 | 代码结构清晰、有 README 与部署文档 | 代码评审通过、文档完整 |

### 1.3 项目定位

| 维度 | 说明 |
|------|------|
| 用户规模 | 个人单用户使用，并发 1-2 |
| 部署形态 | 本地 localhost 运行，不对外发布 |
| 复杂度定位 | MVP，初级开发者可独立完成 |
| 交付定位 | 个人学习成果 + 作品集展示项目 |

## 2. 项目范围

### 2.1 范围内（V1 包含）

| 模块 | 功能点 |
|------|--------|
| 文档上传 | 支持 PDF / Word(docx) / Markdown / TXT；支持批量上传与拖拽上传 |
| 文档解析 | 自动解析文档内容为纯文本；docx 含表格文本；TXT 自动检测编码 |
| 文本切分 | 递归字符切分；Markdown 按标题优先切分；chunk 可配置 |
| 向量化 | Embedding 模型生成向量（OpenAI 与本地模型可配置切换） |
| 向量存储 | Chroma 嵌入式持久化存储向量与元数据 |
| RAG 检索 | 基于问题向量检索 top_k 相关片段 |
| LLM 生成 | 调用大语言模型生成回答，多轮带上下文 |
| 流式输出 | SSE 流式返回回答 |
| 引用来源 | 回答附带引用的文档名与片段 |
| 聊天历史 | 保存会话、问题、回答、引用、耗时、时间戳；支持新建/切换/删除/清空 |
| 文档管理 | 查看已上传文档列表；删除文档（同步删除向量） |

### 2.2 范围外（V1 明确不做）

以下功能**明确不在 V1 范围内**，任何范围变更需经 Human Developer 审批并记录到决策日志：

- OCR（扫描型 PDF 仅友好提示，不处理）
- 多用户与权限管理
- 多知识库（V1 仅单一全局知识库）
- 插件系统、MCP、多 Agent、工作流编排
- 微服务架构、消息队列
- 全文搜索引擎（如 Elasticsearch）
- 暗色模式
- 聊天记录导出
- Rerank 重排序
- Docker 容器化（全项目禁用 Docker）

### 2.3 MVP 范围控制原则

1. **功能闭环优先**：先保证"上传→问答"主链路可用，再优化体验
2. **不过度设计**：扩展点留接口但不实现（如多知识库预留 collection 概念但不做管理 UI）
3. **复杂度预算**：因已选择"多轮对话+流式+双 Embedding"，其余范围严格控制，不再增加复杂特性

## 3. 技术栈概览

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| 后端框架 | Python 3.11+ / FastAPI | 异步、自动生成 OpenAPI 文档 |
| 包管理 | pip + requirements.txt | 初级开发者最易上手 |
| 业务数据库 | MariaDB（本地安装） | 存储文档元数据、聊天历史；**禁用 Docker，本地安装** |
| 向量数据库 | Chroma（嵌入式持久化） | pip 安装，数据落本地 `./data/chroma` |
| Embedding | OpenAI text-embedding-3-small + 本地模型（可配置） | 通过 Provider 抽象层切换 |
| LLM | OpenAI API（gpt-4o-mini 默认，可配置） | 预留兼容其他模型 |
| 文档解析 | PyPDF2 / python-docx / markdown / 前置 chardet | 按文件类型选择解析器 |
| 前端框架 | Vue3 + Vite | SPA |
| UI 组件 | Element Plus | 上传、表格、对话框等 |
| 状态管理 | Pinia | 管理会话、文档列表状态 |
| 路由 | Vue Router | 页面路由 |
| HTTP 客户端 | axios | 调用后端 API |
| 流式接收 | EventSource / fetch streaming | 接收 SSE |

> 技术选型的最终确认由技术负责人（Phase 0）与系统架构师（Phase 1）完成，本表为初步建议。

## 6. 风险登记

| 编号 | 风险项 | 可能性 | 影响 | 缓解措施 | 触发条件 | 应对方案 |
|------|--------|--------|------|----------|----------|----------|
| R1 | 多轮对话上下文管理复杂度超预期 | 高 | 中 | 架构阶段明确上下文截断策略（保留最近 4 轮）；MVP 会话级隔离 | 联调时 Token 超限或上下文丢失 | 降级为单轮独立问答，记录为后续优化 |
| R2 | 双 Embedding 方案增加抽象层复杂度 | 中 | 中 | 架构师设计统一 Provider 接口；默认走 OpenAI，本地模型作为可选 | 本地模型加载失败或效果差 | 暂时只保留 OpenAI 路径，本地模型延后 |
| R3 | 本地 Embedding 模型体积大、首次下载慢 | 中 | 低 | 文档中说明下载步骤；提供模型缓存目录配置 | 用户网络受限无法下载 | 引导使用 OpenAI Embedding 方案 |
| R4 | Chroma 嵌入式模式数据损坏 | 低 | 高 | 定期备份 `./data/chroma`；文档说明备份方法 | 数据加载失败 | 从备份恢复，或重新对文档库做 Embedding |
| R5 | OpenAI API 费用或网络访问问题 | 中 | 高 | 使用 gpt-4o-mini 控制成本；预留 LLM Provider 抽象层支持切换；配置超时重试 | API 调用频繁失败 | 切换至兼容 API（如其他 OpenAI 兼容端点） |
| R6 | 扫描型 PDF 无法解析 | 中 | 低 | 解析字数为 0 时友好提示，不崩溃 | 用户上传扫描件 | 提示"该 PDF 为扫描件，MVP 暂不支持，需 OCR" |
| R7 | SSE 流式输出前后端联调复杂 | 中 | 中 | 后端先用简单文本验证 SSE 通道，前端先 Mock 流式数据 | 流式中断或乱码 | 降级为非流式输出，记录为体验优化项 |
| R8 | 禁用 Docker 导致 MariaDB 环境搭建困难 | 中 | 中 | 提供详细本地安装文档；启动脚本自动检测依赖 | 用户本地 MariaDB 未安装或版本不符 | 文档提供排错指南，启动脚本给出清晰错误提示 |
| R9 | 范围蔓延（开发中临时加功能） | 中 | 中 | 严格变更管理，所有新增需求需审批并记入决策日志 | 用户中途提出新功能 | 评估后纳入 V2 范围，不打乱 V1 节奏 |
| R10 | 初级开发者对 RAG 概念理解不足 | 中 | 中 | AI 工程师在交接文档中补充 RAG 原理说明；关键代码加详细中文注释 | 开发卡壳 | 暂停并请求技术负责人澄清 |

## 10. 变更管理

### 10.1 变更流程

```
提出变更请求
    ↓
项目经理评估影响（范围、工期、风险）
    ↓
记录到决策日志（Proposed 状态）
    ↓
提交 Human Developer 审批
    ↓
通过 → 更新 project-plan.md 与 todo.md → Accepted
拒绝 → 标记 Rejected，归档
```

### 10.2 变更原则

1. 任何范围新增必须经过评估与审批，不得口头默认纳入
2. 已记录的决策不可修改，只能通过新决策推翻（Superseded）
3. 变更影响工期超过 10% 时，需重新规划里程碑
