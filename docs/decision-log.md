#### 背景

#### 决策

采用递归字符切分算法，按段落→句子→字符三级切分。参数：chunk_size=500，overlap=50，最小 chunk=350。

2. 参考 langchain RecursiveCharacterTextSplitter 逻辑
3. Markdown 优先按标题边界切分
4. overlap 保证上下文连贯性

### DEC-20260712-012: 本地 Embedding 模型选用 bge-m3

#### 元信息

| 字段 | 内容 |
|------|------|
| 决策编号 | DEC-20260712-012 |
| 决策日期 | 2026-07-12 |
| 决策者 | Tech Lead |
| 决策状态 | Accepted |
| 决策类型 | 技术选型 |
| 影响范围 | 本地 Embedding Provider 实现 |

#### 背景

DEC-002 确定支持本地 Embedding 模型作为可选方案。需在 bge-m3 与 m3e 之间选择具体模型。两者都是主流中文 Embedding 模型，各有优劣。

#### 问题

本地 Embedding 模型选 bge-m3 还是 m3e？

#### 决策

选用 BAAI/bge-m3 作为本地 Embedding 模型：
- 通过 sentence-transformers 库加载
- 支持密集向量、稀疏向量、多向量检索（MVP 仅用密集向量）
- 维度 1024
- 模型体积约 2.3GB
- 支持中英文，MTEB 中文榜表现优秀

#### 理由

1. bge-m3 多语言能力强，中英文混合文档效果好
2. 智源（BAAI）持续维护，社区活跃
3. sentence-transformers 加载简单，与 Chroma 集成顺畅
4. MTEB 榜单中文表现优于 m3e
5. 维度 1024 与 OpenAI text-embedding-3-small（1536）虽不同，但 Provider 抽象层在切换时重建索引即可

#### 替代方案

| 方案 | 优点 | 缺点 | 放弃原因 |
|------|------|------|----------|
| m3e-large | 纯中文优化、体积小 | 多语言弱、维护不如 bge 活跃 | 中英文混合场景不如 bge-m3 |
| bge-large-zh | 中文优秀 | 仅中文、不支持英文 | 不如 bge-m3 通用 |
| bge-m3（已选择） | 多语言、榜单优、维护活跃 | 体积大（2.3GB） | — |

#### 影响

**正面影响**：本地模型中文检索质量有保障
**负面影响**：
- 首次下载 2.3GB，需文档说明（风险 R3）
- 切换 Embedding 模型需重建向量索引（架构师需在文档中说明）

|------|------|--------|----------|
| 1 | 本地 Provider 实现 bge-m3 加载 | AI Engineer | Phase 3 |
| 2 | 文档说明下载步骤与缓存目录 | DevOps Engineer | Phase 5 |
| 3 | 切换时索引重建逻辑 | AI Engineer | Phase 3 |

### DEC-20260712-011: 多轮对话上下文截断策略

#### 元信息

| 字段 | 内容 |
|------|------|
| 决策编号 | DEC-20260712-011 |
| 决策日期 | 2026-07-12 |
| 决策者 | Tech Lead |
| 决策状态 | Accepted |
| 决策类型 | 设计决策 |
| 影响范围 | RAG 服务、Prompt 工程、Token 成本 |

#### 背景

DEC-003 确定采用多轮带上下文对话。需明确上下文组装与截断策略，避免 Token 超限并控制成本。

#### 问题

多轮对话的上下文如何组装与截断？

#### 决策

采用"系统提示 + 检索上下文 + 最近 N 轮历史"的组装策略：

```
[系统提示 System Prompt]
[检索到的文档片段 RAG Context]  ← 每轮根据当前问题重新检索
[历史对话 History]              ← 保留最近 4 轮（8 条消息）
[当前用户问题]
```

截断规则：
1. 历史对话仅保留最近 4 轮（8 条消息），更早的丢弃
2. 检索上下文 top_k=5 片段，每片段最多 500 字符
3. Token 预算：系统提示约 200 token + 检索上下文约 2000 token + 历史约 1500 token + 当前问题 + 输出预留 1500 token，总计约 6000 token（gpt-4o-mini 上下文 128K，余量充足）
4. 若历史超长则按轮数截断，不按字符截断（保证轮次完整性）
5. 会话级隔离，不同会话上下文独立

#### 理由

1. 最近 4 轮覆盖大多数追问场景，平衡上下文与成本
2. 每轮重新检索保证答案基于最新问题相关内容
3. 按轮数截断保证对话完整性，避免半句话
4. Token 预算远低于模型上限，安全裕度大

#### 替代方案

| 方案 | 优点 | 缺点 | 放弃原因 |
|------|------|------|----------|
| 全部历史 | 上下文完整 | Token 易超限、成本高 | 不安全 |
| 按字符截断 | 精确控制长度 | 可能截断半句话 | 体验差 |
| 最近 4 轮 + 重新检索（已选择） | 平衡、完整 | 个别长对话丢失早期上下文 | — |
| 摘要压缩历史 | 保留更多信息 | 实现复杂、摘要有损 | MVP 不做 |

#### 影响

**正面影响**：上下文管理清晰，成本可控
**负面影响**：超过 4 轮的早期对话不进入上下文（可接受）

#### 相关决策

- DEC-20260712-003: 多轮对话带上下文
- DEC-20260712-006: top_k=5、chunk=500

### DEC-20260712-010: SSE 流式输出技术方案

#### 元信息

| 字段 | 内容 |
|------|------|
| 决策编号 | DEC-20260712-010 |
| 决策日期 | 2026-07-12 |
| 决策者 | Tech Lead |
| 决策状态 | Accepted |
| 决策类型 | 设计决策 |
| 影响范围 | 后端 API、前端、RAG 服务 |

#### 背景

DEC-004 确定采用流式输出 + 引用来源。需明确 SSE 前后端技术实现方案。

#### 问题

SSE 流式输出前后端如何实现？引用来源如何随流式回答传递？

#### 决策

采用 FastAPI StreamingResponse + 前端 fetch streaming 方案：

**后端：**
- 使用 FastAPI `StreamingResponse`，`media_type="text/event-stream"`
- 自定义 SSE 事件格式，区分 token 事件与引用事件：
  - `event: token` `data: {"content": "..."}`
  - `event: references` `data: [{"doc_name": "...", "chunk": "...", "source_path": "..."}]`
  - `event: done` `data: {"message_id": "...", "elapsed_ms": ...}`
  - `event: error` `data: {"message": "..."}`
- 引用来源在流式回答开始前先发送 `references` 事件（基于当前问题检索结果）
- LLM 调用使用 OpenAI SDK 的 stream=True

**前端：**
- 使用 `fetch` + `ReadableStream` 解析 SSE（不用 EventSource，因 EventSource 不支持 POST 与自定义 headers）
- 按事件类型分发：token 追加到回答区，references 渲染到引用区，done 结束加载状态

#### 理由

1. FastAPI 原生支持 StreamingResponse，无需额外依赖
2. 自定义事件格式可区分 token 与引用，前端渲染清晰
3. fetch streaming 比 EventSource 灵活，支持 POST 请求体（需发送问题与会话ID）
4. 引用先于 token 发送，前端可先渲染引用框架

#### 替代方案

| 方案 | 优点 | 缺点 | 放弃原因 |
|------|------|------|----------|
| EventSource | 浏览器原生、简单 | 仅支持 GET、无自定义 header | 无法传 POST body |
| WebSocket | 双向通信 | 过度设计、复杂 | 单向流式无需双向 |
| fetch streaming（已选择） | 灵活、支持 POST | 需手动解析 SSE | — |

#### 影响

**正面影响**：流式体验好，引用与 token 分离清晰
**负面影响**：前端需手动解析 SSE 流（增加约 50 行代码）

#### 实施计划

| 步骤 | 描述 | 负责人 | 预计时间 |
|------|------|--------|----------|
| 1 | 后端 SSE 端点实现 | Backend Engineer | Phase 2 |
| 2 | 前端 fetch streaming 解析 | Frontend Engineer | Phase 3 |
| 3 | 联调验证 | QA Engineer | Phase 4 |

### DEC-20260712-009: LLM Provider 抽象层设计方向

#### 元信息

| 字段 | 内容 |
|------|------|
| 决策编号 | DEC-20260712-009 |
| 决策日期 | 2026-07-12 |
| 决策者 | Tech Lead |
| 决策状态 | Accepted |
| 决策类型 | 设计决策 |
| 影响范围 | AI 服务模块、LLM 调用 |

#### 背景

风险 R5（OpenAI API 费用或网络问题）要求预留 LLM Provider 抽象层，支持切换至兼容端点。需明确抽象层设计方向。

#### 问题

LLM Provider 抽象层如何设计以支持 OpenAI 与兼容端点切换？

#### 决策

设计 LLM Provider 抽象层：

```python
class LLMProvider(ABC):
    @abstractmethod
    async def chat_completion(
        self,
        messages: List[Message],
        stream: bool = False,
        **kwargs
    ) -> Union[ChatResponse, AsyncIterator[ChatChunk]]:
        """对话补全，支持流式与非流式"""
        pass

    @abstractmethod
    async def embed(self, texts: List[str]) -> List[List[float]]:
        """文本向量化（复用于 Embedding，或独立）"""
        pass
```

实现策略：
- `OpenAILLMProvider`：基于 openai SDK，通过 `base_url` 与 `api_key` 配置支持任意 OpenAI 兼容端点（如 Azure OpenAI、第三方代理、本地 vLLM）
- 默认 `base_url=https://api.openai.com/v1`，可通过 `.env` 的 `LLM_BASE_URL` 覆盖
- 模型名通过 `LLM_MODEL` 配置（默认 gpt-4o-mini）
- 工厂函数 `get_llm_provider()` 根据 `.env` 返回实例

#### 理由

1. OpenAI SDK 原生支持 `base_url` 参数，兼容端点无需额外开发
2. 抽象层为后续支持非 OpenAI 协议模型（如 Anthropic）留接口
3. 配置切换简单，只需改 `.env`
4. 复用 openai SDK 减少依赖

#### 替代方案

| 方案 | 优点 | 缺点 | 放弃原因 |
|------|------|------|----------|
| 直接调用 openai SDK | 最简 | 无法切换、耦合 | 不符合风险 R5 缓解 |
| langchain 封装 | 功能丰富 | 重、学习成本高 | MVP 不引入大框架 |
| 自定义抽象层（已选择） | 轻量、灵活 | 需自行维护 | — |

#### 影响

**正面影响**：LLM 切换灵活，风险 R5 有效缓解
**负面影响**：增加少量抽象代码（约 100 行）

### DEC-20260712-008: Embedding Provider 抽象层设计方向

#### 背景

DEC-002 确定支持 OpenAI + 本地模型双 Embedding 方案。需明确 Provider 抽象层设计方向，供架构师与 AI 工程师实现。

#### 问题

Embedding Provider 抽象层如何设计以支持配置切换？

#### 决策

设计 Embedding Provider 抽象层：

```python
class EmbeddingProvider(ABC):
    @abstractmethod
    async def embed(self, texts: List[str]) -> List[List[float]]:
        """文本向量化"""
        pass

    @property
    @abstractmethod
    def dimension(self) -> int:
        """向量维度"""
        pass

    @property
    @abstractmethod
    def name(self) -> str:
        """Provider 名称，用于 Chroma collection 标识"""
        pass
```

实现策略：
- `OpenAIEmbeddingProvider`：text-embedding-3-small，维度 1536
- `LocalBgeM3Provider`：bge-m3，维度 1024（DEC-012）
- 工厂函数 `get_embedding_provider()` 根据 `.env` 的 `EMBEDDING_PROVIDER`（openai/local）返回实例
- Chroma collection 命名包含 provider 与维度信息（如 `kb_openai_1536`），切换 provider 时使用新 collection，避免维度不匹配
- 切换 provider 后需对文档库重新 Embedding（文档中说明）

#### 理由

1. 抽象层统一接口，AI 工程师实现双 Provider
2. dimension 属性让系统能感知向量维度，避免 Chroma 维度冲突
3. collection 命名含 provider 信息，隔离不同 provider 数据
4. 工厂模式配置切换简单

#### 替代方案

| 方案 | 优点 | 缺点 | 放弃原因 |
|------|------|------|----------|
| 直接调用，无抽象 | 最简 | 切换需改代码 | 不符合 DEC-002 |
| 单一 Provider + 配置参数 | 简单 | 无法支持本地模型 | 不符合 DEC-002 |
| 抽象层 + 工厂（已选择） | 灵活、可扩展 | 代码量增加 | — |

#### 影响

**正面影响**：双 Provider 切换清晰，维度冲突可避免
**负面影响**：
- 切换 provider 需重建索引（文档需说明）
- 增加抽象代码（约 150 行）

### DEC-20260712-002: Embedding 采用 OpenAI + 本地模型双方案可配置

#### 元信息

| 字段 | 内容 |
|------|------|
| 决策编号 | DEC-20260712-002 |
| 决策日期 | 2026-07-12 |
| 决策者 | Human Developer |
| 决策状态 | Accepted |
| 决策类型 | 技术选型 |
| 影响范围 | 向量化模块、AI 服务架构、依赖管理 |

#### 背景

Embedding 模型选择影响中文检索效果、API 成本与依赖复杂度。OpenAI text-embedding-3-small 配置简单但有 API 费用；本地中文模型（如 bge-m3、m3e）中文效果好且免费但需下载大模型文件。项目经理推荐 OpenAI 单方案以控制 MVP 复杂度，Human Developer 选择两者都支持以获得更好扩展性。

#### 问题

Embedding 向量化模型用哪种方案？

#### 决策

采用双方案可配置：
- 设计 Embedding Provider 抽象层（统一接口）
- 默认走 OpenAI text-embedding-3-small
- 支持通过配置切换为本地中文模型（bge-m3 或 m3e）
- 通过 `.env` 配置项切换

#### 理由

1. 双方案兼顾成本与中文效果
2. Provider 抽象层为后续扩展其他模型留接口
3. 默认 OpenAI 保证开箱即用，本地模型作为可选
4. Human Developer 主动选择，愿意承担抽象层复杂度

#### 替代方案

| 方案 | 优点 | 缺点 | 放弃原因 |
|------|------|------|----------|
| 仅 OpenAI | 配置简单、效果好 | API 费用、中文非最优 | Human Developer 选择双方案 |
| 仅本地模型 | 免费、中文优 | 模型体积大、首次下载慢 | 失去 OpenAI 便捷性 |
| 双方案可配置（已选择） | 扩展性好、灵活 | 抽象层复杂度高 | — |

#### 影响

**正面影响**：
- 扩展性好，支持成本与效果灵活取舍
- Provider 抽象层为后续模型扩展打基础

**负面影响**：
- 增加抽象层设计与测试工作量
- 本地模型首次下载体积大（几百 MB~1GB）
- 风险 R2（抽象层复杂度）、R3（模型下载慢）

**受影响方**：
- 系统架构师：设计 Provider 抽象接口
- AI 工程师：实现双 Provider
- 测试工程师：需测试两种 Provider

#### 实施计划

| 步骤 | 描述 | 负责人 | 预计时间 |
|------|------|--------|----------|
| 1 | 设计 Embedding Provider 抽象接口 | Solution Architect | Phase 1 |
| 2 | 实现 OpenAI Provider（默认） | AI Engineer | Phase 3 |
| 3 | 实现本地模型 Provider | AI Engineer | Phase 3 |
| 4 | 配置切换机制 | AI Engineer | Phase 3 |

### DEC-20260712-001: 项目采用本地 localhost 运行，无需认证

#### 元信息

| 字段 | 内容 |
|------|------|
| 决策编号 | DEC-20260712-001 |
| 决策日期 | 2026-07-12 |
| 决策者 | Human Developer |
| 决策状态 | Accepted |
| 决策类型 | 运维决策 / 架构决策 |
| 影响范围 | 部署形态、认证机制、环境配置、安全策略 |

#### 背景

项目定位为个人学习与作品集展示，需要确定部署与访问形态。本地运行最简单但无法在线演示；公网部署便于展示但需认证与 HTTPS。项目经理推荐本地运行以匹配 MVP 定位。

#### 问题

项目的部署与访问形态如何确定？

#### 决策

采用本地 localhost 运行，无需认证：
- 应用仅绑定 localhost，不对外发布
- 无需用户登录与权限管理
- 无需 HTTPS（本地明文可接受）
- 作品集通过截图 + 录屏展示

#### 理由

1. 匹配 MVP 定位与个人单用户使用场景
2. 无需认证简化 API 设计与前端流程
3. 无需 HTTPS 简化证书管理
4. 本地运行安全风险最低

#### 替代方案

| 方案 | 优点 | 缺点 | 放弃原因 |
|------|------|------|----------|
| Docker Compose 本地部署 | 环境隔离 | 需学 Docker（与 DEC-005 冲突） | 已被 DEC-005 禁止 |
| 公网部署 + 访问令牌 | 在线演示 | 需认证、HTTPS、防滥用 | 复杂度过高，不符合 MVP |
| 本地 localhost 运行（已选择） | 最简、最安全 | 无法在线演示 | — |

#### 影响

**正面影响**：
- 部署最简单，启动脚本即可
- 无需认证模块，减少开发量
- 安全面最小

**负面影响**：
- 无法在线演示，作品集需截图录屏
- API Key 仍需通过 `.env` 安全管理

**受影响方**：
- 运维工程师：部署方案聚焦本地启动
- 安全工程师：重点转为 API Key 管理与文件上传安全
- 前端工程师：无需登录页

#### 相关决策

- DEC-20260712-005: 全项目禁用 Docker（进一步约束部署方案）

|------|----------|------|------|
